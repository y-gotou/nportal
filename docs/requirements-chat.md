# 会議チャット機能 要件定義書

- 作成日: 2026-06-12
- 更新日: 2026-06-12(UIヒアリング結果を反映)
- ステータス: ドラフト(ヒアリング完了・実装着手前)

## 1. 背景・目的

社内 AI 勉強会の会議は Teams で実施しているが、共通アカウントで運用しているため、会議中に参加者個人が資料や URL、画像を共有することが難しい。N Portal 上に会議ごとのチャット欄を設け、会議中に文章・URL・画像・ファイルをリアルタイムに共有できるようにする。

## 2. スコープ

### 対象

- 会議(スケジュール)ごとのチャットルーム
- テキスト・URL・画像・ファイルの投稿
- リアクション・スタンプ・引用返信
- 過去会議のチャットログ閲覧(読み取り専用)

### 対象外

- 常設(会議に紐づかない)チャットルーム
- スレッド形式の返信
- Markdown 記法・コードブロック装飾
- URL の OGP プレビュー、未読バッジ
- 投稿の編集機能(本人・管理者とも)
- 匿名投稿、表示名のカスタマイズ

## 3. 機能要件

### 3.1 チャットルーム

| 項目 | 内容 |
| --- | --- |
| ルーム単位 | `schedule` テーブルの会議 1 件につき 1 ルーム |
| 入り口 | ①スケジュールページの各会議カードに「チャット」ボタンを追加 ②トップページのヒーローカード(次回の会議)に「チャットを開く」ボタンを追加。ルーム一覧専用ページ・ナビメニューは作らない |
| 投稿可能期間 | 会議日当日まで投稿可能。**会議日の翌日以降は自動で読み取り専用**(クローズ操作は不要) |
| ログ保存 | 永久保存。過去会議のログは読み取り専用で全ユーザーが閲覧可能 |

### 3.2 メッセージ投稿

| 種別 | 内容 |
| --- | --- |
| テキスト | プレーンテキスト。本文中の URL は自動でクリック可能なリンクに変換する |
| 画像 | PNG / JPEG / GIF / WebP をアップロードし、チャット内にインラインプレビュー表示 |
| ファイル | 既存の資料共有と同じ許可形式(PDF / Office / CSV / TXT / MD / HTML 等)。ダウンロードリンク形式で表示 |
| スタンプ | プリセット絵文字(👍🎉😆 等・数十種)を選び、大きめ表示の単独メッセージとして投稿 |
| 引用返信 | 既存の投稿を引用して返信できる(スレッドではなく引用ブロック+本文の形式) |

- ファイル・画像のサイズ上限は資料共有と同一: **50MB**(`MAX_RESOURCE_FILE_SIZE` を共用)
- 許可拡張子・MIME 判定も資料共有の `MIME_TYPES_BY_EXTENSION` を共用する(zip は管理者のみ、の既存ルールも踏襲)
- 添付ファイルの保存先は既存の R2 バケット `RESOURCES_BUCKET`(チャット用プレフィックスで分離)

### 3.3 リアクション

- 各投稿に対して絵文字リアクション(👍 等のプリセット)を付与・解除できる
- 同一ユーザー × 同一絵文字 × 同一投稿は 1 回まで。リアクション数と自分が押したかを表示する
- 読み取り専用になった過去ログにはリアクションも付与できない

### 3.4 削除

| 操作者 | 可能な操作 |
| --- | --- |
| 投稿者本人 | 自分の投稿の削除(投稿可能期間中のみ) |
| 管理者(`ADMIN_EMAILS`) | 全投稿の削除。**過去会議の読み取り専用ログに対しても削除可能** |

- 編集機能は設けない(誤投稿は削除して再投稿)
- 添付ファイル付き投稿を削除した場合、R2 上のオブジェクトも削除する
- 削除された投稿を引用していた返信は「削除された投稿」と表示する

### 3.5 投稿者表示

- Cloudflare Access の JWT から取得したメールアドレスをそのまま投稿者名として表示する(資料共有の `submitted_by` と同様)
- 表示名設定・匿名投稿は行わない

### 3.6 リアルタイム更新

- **ポーリング方式**: チャットページ表示中、2〜3 秒間隔で新着を確認する
- **バージョン番号方式による通信量最小化**(会議室のネットワークが弱いため):
  - ルームごとにバージョン番号を保持し、投稿・削除・リアクションのいずれかの変更で増加させる
  - クライアントは手元のバージョンを付けてポーリングし、**変更がなければサーバーは `204 No Content`(ボディ 0 バイト)を返す**
  - 変更があった場合のみ、新着メッセージ・リアクション・削除 ID の差分を返す
  - アイドル時の通信は HTTP ヘッダーのみ(往復数百バイト程度)
- ページ非表示時(タブ非アクティブ)はポーリングを停止または間引きする
- WebSocket / Durable Objects は採用しない(構成の複雑化を避けるため。ヒアリングにて合意済み)

## 4. 非機能要件

| 項目 | 内容 |
| --- | --- |
| 認証 | 既存の Cloudflare Access 必須(`/chat` 配下と関連 API を `server/middleware/auth.ts` の認証必須パスに追加) |
| 想定同時利用者 | 勉強会参加者規模(〜20 名程度)。D1 へのポーリング負荷はこの規模で問題ない設計とする |
| ネットワーク | 会議室の回線が弱い前提。アイドル時のポーリング通信を最小化する(バージョン番号方式・204 応答) |
| インフラ | 既存構成のまま(Nuxt 4 + Cloudflare Pages Functions + D1 + R2)。追加インフラなし |
| データ保持 | チャットログ・添付ファイルは無期限保存 |
| XSS 対策 | 投稿本文はエスケープして表示。URL リンク化は `http(s)://` のみ対象 |

## 5. データモデル案(D1)

```sql
CREATE TABLE chat_messages (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  schedule_id INTEGER NOT NULL REFERENCES schedule(id),
  user_email  TEXT NOT NULL,
  kind        TEXT NOT NULL DEFAULT 'text',  -- 'text' | 'stamp'
  body        TEXT NOT NULL DEFAULT '',       -- テキスト本文 or スタンプ絵文字
  reply_to_id INTEGER REFERENCES chat_messages(id),
  file_key    TEXT,                           -- R2 オブジェクトキー(添付時)
  file_name   TEXT,
  file_size   INTEGER,
  mime_type   TEXT,
  deleted_at  TEXT,                           -- 論理削除(引用表示のため)
  created_at  TEXT DEFAULT (datetime('now'))
);

CREATE TABLE chat_reactions (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  message_id INTEGER NOT NULL REFERENCES chat_messages(id),
  user_email TEXT NOT NULL,
  emoji      TEXT NOT NULL,
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(message_id, user_email, emoji)
);

CREATE TABLE chat_room_state (
  schedule_id INTEGER PRIMARY KEY REFERENCES schedule(id),
  version     INTEGER NOT NULL DEFAULT 0,
  updated_at  TEXT DEFAULT (datetime('now'))
);

CREATE INDEX idx_chat_messages_schedule ON chat_messages(schedule_id, id);
CREATE INDEX idx_chat_reactions_message ON chat_reactions(message_id);
```

- 引用返信の表示整合性のため、メッセージは論理削除(`deleted_at`)とし、添付ファイルの R2 オブジェクトは物理削除する
- `chat_room_state.version` は投稿・削除・リアクションのすべての変更操作で +1 する(ポーリングの変更検知用)

## 6. API 案

| メソッド / パス | 内容 |
| --- | --- |
| `GET /api/chat/[scheduleId]/messages?after=<id>&version=<v>` | ポーリング用。ルームのバージョンが `version` と同じなら **`204 No Content`(ボディなし)**。変更があれば新着メッセージ・全リアクション・削除済み ID・新バージョン・会議情報(読み取り専用判定含む)を返す。初回は `version` なしで全件取得 |
| `POST /api/chat/[scheduleId]/messages` | 投稿(テキスト / スタンプ / 添付付き)。読み取り専用ルームは 403 |
| `DELETE /api/chat/messages/[id]` | 投稿削除(本人 or 管理者) |
| `PUT /api/chat/messages/[id]/reactions` | リアクション付与・解除 |

- 読み取り専用判定はサーバー側で必ず行う(`schedule.date < 今日` なら投稿・リアクション・本人削除を拒否。管理者削除のみ許可)

## 7. UI 仕様(ヒアリング確定)

### 7.1 入り口

- **スケジュールページ** (`/schedule`): 「今後の会議」「過去の会議」両方の各会議カードのボタン欄(「議事録を見る」と同列)に「チャット」ボタンを追加
- **トップページ** (`/`): ヒーローカード(次回の会議)のアクション欄(「会議に参加する」「予定一覧を見る」と同列)に「チャットを開く」ボタンを追加
- ルーム一覧専用ページ(`/chat`)とナビの「チャット」メニューは作らない

### 7.2 チャットルーム画面(`/chat/[scheduleId]`)

- 上部: 会議タイトル・日時、読み取り専用バッジ(過去会議の場合)
- 中央: メッセージ一覧、下部: 入力欄(読み取り専用時は非表示)

**メッセージ一覧 — LINE 風レイアウト**

- 自分の投稿は右側、他人の投稿は左側の吹き出しで表示
- 他人の投稿には投稿者名(メールアドレス)を吹き出し上部に表示
- 時刻は吹き出しの脇に表示
- 新着メッセージで自動スクロール
- 引用返信は吹き出し内に引用ブロック+本文の形式
- スタンプ(絵文字)は吹き出しなし・大きめ表示

**操作メニュー — ホバーでアイコン表示**

- メッセージにマウスを乗せると操作アイコンが浮かぶ:
  `👍 😀 😮 😢 | その他リアクション | 返信 | 削除`
- 先頭の 4 つはクイックリアクション(1 クリックで付与)
- 「その他リアクション」で絵文字ピッカーを開き、プリセット全種から選択
- 「削除」は本人の投稿(または管理者)のみ表示
- アイコンは **Lucide**(`lucide-vue-next`)を導入して使用する

**入力欄**

- テキスト入力: **Enter で送信、Shift+Enter で改行**(日本語 IME の変換確定 Enter では送信しない)
- 添付ボタン(画像・ファイル)+ スタンプピッカー + 送信ボタン

**画像表示**

- チャット内はサムネイル表示、クリックで画面内モーダル(オーバーレイ)拡大表示

### 7.3 対応環境

- PC 主体。スマホはレイアウトが崩れない程度のレスポンシブ対応とし、ホバー操作のタッチ代替(長押し等)の作り込みは最小限とする

## 8. 決定事項サマリ(ヒアリング結果)

| 論点 | 決定 |
| --- | --- |
| 更新方式 | ポーリング(2〜3 秒)+バージョン番号方式(変更なしなら 204 応答でアイドル時通信を最小化)。Durable Objects は不採用 |
| ルーム単位 | 会議(スケジュール)ごと |
| 投稿種別 | テキスト+URL 自動リンク / 画像 / ファイル / スタンプ(大きめ絵文字) |
| 投稿者名 | メールアドレスを自動表示 |
| 編集・削除 | 編集は不可。削除は本人+管理者(過去ログは管理者のみ) |
| 追加機能 | リアクション、引用返信、スタンプ。OGP プレビュー・未読バッジは見送り |
| サイズ上限 | 資料共有と同じ 50MB・同じ許可形式 |
| ログ | 永久保存。会議日の翌日以降は自動で読み取り専用 |
| UI | 専用チャットページ(全画面)。入り口はスケジュールページ+トップのヒーローカード |
| レイアウト | LINE 風(自分右・他人左の吹き出し) |
| 送信操作 | Enter 送信 / Shift+Enter 改行(IME 確定 Enter は除外) |
| 操作メニュー | ホバーで「👍😀😮😢/その他リアクション/返信/削除」アイコン表示。アイコンは Lucide を導入 |
| 画像クリック | モーダルで拡大表示 |
| 対応環境 | PC 主体。スマホは崩れない程度のレスポンシブ対応 |
